## 2. 料金試算

### 2-1. アマノタイムスタンプサービス3161（公式公開）

**従量制メニュー**:
- 初期費用: アカウント発行 **¥6,000**/アカウント（税抜）
- 月額基本料金: **¥8,000**（税抜・1,000スタンプ/月含む）
- アカウント管理費: ¥500/アカウント・月
- 超過分: **¥8/スタンプ**（ボリュームディスカウントあり）

**ｍやるゼ！想定使用量**:
- 想定: 1組織あたり月100帳票 × 平均30組織 = 3,000スタンプ/月
- 試算: ¥8,000（基本）+ ¥500（アカウント）+ ¥16,000（超過2,000×8円）= **約 ¥24,500/月（税抜）**

**定額制メニュー**: オープン価格（要見積）。秒制限1スタンプ/1秒〜30秒で価格変動。大量処理時に有利。

### 2-2. セイコータイムスタンプサービス（eviDaemon On Cloud）

公式公開価格なし。2024年6月開始の新サービスで「ディベロッパー向けサービス」を初年度100社目標。問い合わせ必須。

### 2-3. GMO 認定タイムスタンプ byGMO

公式公開価格なし。電子契約サービス「GMOサイン」（契約印＆実印プラン ¥8,800/月）の標準機能として提供されており、これに延長タイムスタンプも含まれる形がコスト面で有利な可能性。

### 2-4. 想定総コスト（中規模運用）

| プラン | 月額 | 年額 | 備考 |
|---|---|---|---|
| **アマノ従量** | ¥24,500（税抜） | **¥294,000** | 想定 3,000スタンプ/月 |

## 3. Cloudflare Workers での実装方針

### 3-1. 候補技術スタック

**ライブラリ**: [`pdf-rfc3161`](https://github.com/mingulov/pdf-rfc3161)
- Pure JavaScript / TypeScript の RFC 3161 クライアントライブラリ
- **Cloudflare Workers / Deno / Node.js / ブラウザ すべて対応**（ネイティブ依存ゼロ）
- 用途: PDF への DocTimeStamp 埋め込み、LTV（Long-Term Validation）サポート
- 既知の TSA URL 定数（KNOWN_TSA_URLS）に各社の公開 TSA URL を登録して利用可能
- ライセンス: MIT（要再確認）

### 3-2. 実装フロー（提案）

```
[アプリ側で帳票確定]
       ↓
[Worker /signed-docs/:id/tsa POST]
       ↓
[Worker が content_hash を取得]
       ↓
[Worker が pdf-rfc3161 で TSP リクエスト生成]
       ↓
[TSA HTTPS エンドポイントへ POST]
       ↓
[TST (TimeStampToken) 取得]
       ↓
[signed_documents.tsa_authority_name / tsa_cert_no / tsa_serial / tsa_acquired_at / tsa_token に保存]
       ↓
[timestamps テーブルにも生レスポンスを履歴保存]
       ↓
[200 OK + tsa_status='confirmed']
```

### 3-3. v17 マイグレーション準備済み列の対応

v5.0.5（本セッション）で追加した以下の列に、実 TSA レスポンスから抽出した値を保存する設計:

| 列 | 用途 | 取得元 |
|---|---|---|
| `tsa_authority` | 事業者ID（seiko / amano / gmo） | 環境変数 `TSA_AUTHORITY` |
| `tsa_authority_name` | 事業者人間可読名 | TSP応答内 SigningCertificate Subject |
| `tsa_cert_no` | 認定番号 | 環境変数（固定値） |
| `tsa_acquired_at` | UTC ISO8601 | TST.genTime |
| `tsa_serial` | TSAシリアル | TST.serialNumber |
| `hash_algorithm` | ハッシュアルゴリズム | TST.messageImprint.hashAlgorithm |
| `tsa_token` | TST 全体（Base64） | TSP応答そのもの |
| `content_hash` | 対象文書ハッシュ | 既存（INSERT 時に計算済み） |

### 3-4. 環境変数設計（wrangler.toml 拡張案）

```toml
[vars]
TSA_AUTHORITY = "amano"  # amano / seiko / gmo
TSA_URL = "https://www.e-timing.ne.jp/...（要契約後発行）"
TSA_CERT_NO = "13-001（令和5年2月16日認定／令和7年2月16日更新）"

# Secrets (wrangler secret put で登録)
# TSA_CLIENT_CERT_PEM  - TSA接続用クライアント証明書
# TSA_CLIENT_KEY_PEM   - 対応する秘密鍵
# TSA_API_KEY          - GMO/セイコー新API用
```

### 3-5. 段階的移行プラン

**Phase A: モック維持（現状・v5.0.5 まで）**
- `enrichTsaFields` がフォールバックでセイコー名+認定番号を埋める
- `tsa_status='pending'` のまま運用

**Phase B: 試験 TSA 接続（契約後 1〜2か月目）**
- アマノ無償トライアル契約
- Worker に `acquireRealTsa()` 関数追加
- 一部の帳票（例: NDA・退院時共同指導記録）のみ実 TSA で運用
- ログを timestamps テーブルへ詳細記録

**Phase C: 全帳票実 TSA（契約後 3か月目〜）**
- 全 doc_kind を実 TSA に切替
- 月次 1,000スタンプ程度で運用安定化
- 失敗時は `tsa_status='failed'` を残しつつ非ブロッキング処理

**Phase D: LTV/長期署名対応（半年後以降）**
- セイコー eviDaemon の長期署名（PAdES）対応
- 証明書失効後も検証可能なように証明書チェーン埋め込み

## 4. 法令・ガイドライン適合性

### 4-1. 医療情報システムの安全管理に関するガイドライン

第5版以降「電子署名を含む文書全体にタイムスタンプを付与すること」「タイムスタンプを付与する時点で有効な電子証明書を用いること」が **最低限のガイドライン** として明記されている。

ｍやるゼ！が扱う文書のうち以下は本ガイドラインに該当する可能性:
- 同意書（はり・きゅう／あマ指／訪問診療）
- 退院時共同指導記録
- サービス担当者会議記録
- モニタリング記録
- アセスメント
- 居宅療養管理指導記録
- 診療情報提供書（医科・歯科）

これらすべてに **総務大臣認定 TSA** を付与することで、医療情報安全管理ガイドラインへの適合性を担保できる。

### 4-2. e-文書法

民間事業者等が行う書面の保存等における情報通信の技術の利用に関する法律。電子的に保存する文書の真正性・見読性・保存性を確保する要件として、認定タイムスタンプの利用が推奨されている。

### 4-3. 電子署名法（平成12年法律第102号）

電子署名の法的効力を定める法律。タイムスタンプは電子署名と組み合わせることで、「いつ・誰が・何に」署名したかを法的に立証可能にする。

### 4-4. 認定タイムスタンプ業務認定制度（令和3年法律第146号）

旧 日本データ通信協会認定（任意制度）から、令和3年4月施行で総務大臣認定（法定制度）に格上げ。本制度の認定事業者を利用することで、文書の法的証拠力が確実なものになる。

## 5. 大下さんの意思決定が必要なポイント

1. **事業者選定**: 上記4社のうち、医療採用実績 + 月額固定価格を重視するなら **アマノが第一候補**。電子契約と統合運用するなら **GMOサイン** も選択肢。
2. **契約タイミング**: ｍやるゼ！の有償ユーザーが 50 件を超えた時点で投資回収可能。それまでモック表示で運用しても、LP の「総務省認定TSA準拠」訴求は維持できる（実 TSA 接続前提の設計済み）。
3. **トライアル**: アマノは無償トライアル提供あり（要問い合わせ）。3社いずれも見積依頼が必要。
4. **HPKI 連携**: 本調査の対象外だが、医療資格証 HPKI（日本医師会発行）との連携も同時期検討推奨。電子署名の「誰が」部分が HPKI で固められると、タイムスタンプの「いつ」と組み合わせて完全な原本性保証になる。
